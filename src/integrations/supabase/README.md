# Supabase Setup (Option B — External Project)

This app talks to the **external** Supabase project `pigisgebfcbfvvflxkdw`,
NOT the Lovable-managed project. Follow these rules to keep it that way.

## Client-side code

Always import from one of:

```ts
import { supabase } from "@/integrations/supabase/own-client"; // canonical
import { supabase } from "@/integrations/supabase/client"; // re-exports own-client
```

Both point at `pigisgebfcbfvvflxkdw`.

## Server-side code (`createServerFn`)

Do NOT use `auth-middleware.ts` or `client.server.ts` as-is — they are
auto-generated by Lovable and point at the **managed** project.

Build a client inline from the `MY_SUPABASE_*` secrets instead:

```ts
import { createClient } from "@supabase/supabase-js";

const supabaseAdmin = createClient(
  process.env.MY_SUPABASE_URL!,
  process.env.MY_SUPABASE_SERVICE_ROLE_KEY!,
);
```

Available secrets for the external project:

- `MY_SUPABASE_URL`
- `MY_SUPABASE_ANON_KEY`
- `MY_SUPABASE_SERVICE_ROLE_KEY`

## Schema changes

The Lovable migration tool targets the managed project, so it will NOT
affect `pigisgebfcbfvvflxkdw`. Workflow:

1. Make schema changes in the Supabase dashboard for `pigisgebfcbfvvflxkdw`.
2. Update the hand-maintained types in `src/integrations/supabase/db.ts`
   (and re-export new enums/row types from there).
3. App code imports types from `db.ts`, never from the auto-generated
   `types.ts` (which reflects the empty managed project).

## Regeneration caveat

Lovable may rewrite these files at any time:

- `client.ts`, `client.server.ts`
- `auth-middleware.ts`, `auth-attacher.ts`
- `types.ts`

If TypeScript errors or wrong-DB writes suddenly reappear, re-apply the
overrides documented here.
