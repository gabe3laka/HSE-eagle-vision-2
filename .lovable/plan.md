## Caveats (important)

- **Auth passwords can't be exported.** Existing Lovable Cloud users will be re-created with the same UUIDs and emails, but they must use "forgot password" on first login. Their profile/incidents/sessions stay linked because UUIDs are preserved.
- **Google OAuth must be re-enabled in your Supabase dashboard** (Auth → Providers → Google). Lovable's broker only works with Lovable Cloud, so we'll fall back to standard Supabase Google OAuth, which needs a Google Cloud OAuth client ID/secret you create.
- **`types.ts` becomes manual.** It's currently auto-regenerated by Lovable Cloud. After migration, schema changes you make in your Supabase dashboard won't auto-update `types.ts` — we already have `src/integrations/supabase/db.ts` as a typed shim that mitigates this.
- **Lovable Cloud stays provisioned** (can't be disabled), but unused. The Cloud panel's migration/linter tools won't reflect your Supabase.

## Plan

### 1. Collect secrets
I request `MY_SUPABASE_URL`, `MY_SUPABASE_ANON_KEY`, `MY_SUPABASE_SERVICE_ROLE_KEY`. You paste values. Implementation pauses until this is done.

### 2. Apply schema to your Supabase
A Node script (run via `code--exec`) connects to your Supabase using the service role key and creates everything currently in Lovable Cloud:
- Enums: `hazard_type`, `severity`, `session_status`
- Tables: `profiles`, `alert_settings`, `monitoring_sessions`, `hazard_zones`, `detections`, `incidents`
- All `GRANT`s + RLS policies scoped to `auth.uid()`
- `handle_new_user` trigger on `auth.users` → auto-creates a `profiles` row
- Storage bucket `incident-snapshots` with its RLS policies

### 3. Migrate data
Same script, FK-safe order:
1. Read from Lovable Cloud via `supabase--read_query`
2. Re-create auth users in target (UUIDs preserved, emails copied, no passwords) via Admin API
3. Bulk-insert `profiles`, `alert_settings`, `hazard_zones`, `monitoring_sessions`, `detections`, `incidents` using service role (RLS bypassed)
4. Copy storage objects from `incident-snapshots` (download via signed URL → re-upload, same paths)

### 4. Rewire the app client
- New file `src/integrations/supabase/own-client.ts` reading `VITE_MY_SUPABASE_URL` / `VITE_MY_SUPABASE_ANON_KEY`
- New file `src/integrations/supabase/own-client.server.ts` reading `MY_SUPABASE_URL` / `MY_SUPABASE_SERVICE_ROLE_KEY`
- Update `src/integrations/supabase/db.ts` and `src/contexts/AuthContext.tsx` to import from `own-client.ts` instead of `client.ts`
- Find/replace remaining `@/integrations/supabase/client` imports across hooks, pages, components to point at the new client
- Leave the auto-generated `client.ts` / `client.server.ts` files untouched (Lovable keeps regenerating them; we just stop importing them)

### 5. Configure your Supabase auth
- Enable email/password (you do this in your Supabase dashboard, one click)
- Send password-reset emails to migrated users from the script (optional, on your go-ahead)
- (Optional) You enable Google OAuth in Supabase dashboard with your own Google Cloud client ID/secret; I'll wire `signInWithOAuth("google")` in `src/pages/Auth.tsx`

### 6. Verify
- Sign up a fresh test user → confirm `profiles` row auto-creates in **your** Supabase
- Start a monitoring session → row lands in your `monitoring_sessions`
- Trigger an incident → snapshot uploads to your `incident-snapshots` bucket
- Sign in as a migrated user via password-reset flow

## Technical notes

- All UUIDs (`id`, `user_id`, FKs) copied verbatim — relational integrity preserved.
- Inserts run under service role so RLS doesn't block migration.
- Storage migration uses signed URLs (60s TTL) for source reads, `storage.upload` with `upsert: true` on target.
- `db.ts` continues to cast `supabase as any` for `.from(...)`, so missing auto-generated types won't break the build.
- `.env` is auto-managed by Lovable Cloud — I won't edit it. The new `VITE_MY_*` vars come from the secrets you set, which Lovable injects at build/runtime.
