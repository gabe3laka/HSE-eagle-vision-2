# Migrate to Your Own Supabase

You've connected your Supabase account. Here's how we move the Eagle Vision backend off Lovable Cloud onto a project you own and bill for.

## Important caveats (please read)

- **Lovable Cloud can't be disabled.** It will remain provisioned on this project but unused. Future migrations/tools from the Cloud panel won't apply to your Supabase — we'll manage your project from your own Supabase dashboard.
- **Auth users can't be perfectly migrated.** Supabase doesn't allow exporting password hashes between projects via the API. Existing users will need to **reset their password** (or re-sign-up) on first login. Profile/incident data tied to their `user_id` will be preserved if we re-create them with the same UUID, or remapped otherwise.
- **The generated `client.ts` / `types.ts` files are auto-managed by Lovable Cloud.** We'll need to bypass them and use a hand-written client pointed at your Supabase URL + anon key.

## Plan

### 1. Confirm target Supabase project
Pick which connected Supabase project (from your connector) is the target. I'll grab its `URL`, `anon key`, and `service_role key`.

### 2. Apply schema to your Supabase
Re-run the full schema on your project:
- Tables: `profiles`, `alert_settings`, `monitoring_sessions`, `hazard_zones`, `detections`, `incidents`
- RLS policies (scoped to `auth.uid()`)
- `handle_new_user` trigger that auto-creates a profile row on signup
- Storage bucket `incident-snapshots` + its RLS policies

### 3. Migrate data
Export from Lovable Cloud → import into your Supabase, in FK-safe order:
1. `auth.users` (emails + UUIDs only — passwords not portable; we'll trigger password-reset emails)
2. `profiles`
3. `alert_settings`, `hazard_zones`
4. `monitoring_sessions`
5. `detections`, `incidents`
6. Storage objects in `incident-snapshots` (download + re-upload)

I'll do this with `pg_dump`/`COPY` via the service role keys on both sides.

### 4. Rewire the app client
- Add secrets `OWN_SUPABASE_URL`, `OWN_SUPABASE_ANON_KEY`, `OWN_SUPABASE_SERVICE_ROLE_KEY` (the connector may already inject these — I'll verify).
- Replace `src/integrations/supabase/client.ts` consumption with a new `src/integrations/supabase/own-client.ts` that reads from `import.meta.env.VITE_OWN_SUPABASE_URL` / `..._ANON_KEY`.
- Update `client.server.ts` equivalent for server functions to use your service role.
- Update `src/integrations/supabase/db.ts` typed accessor to import from the new client.
- Regenerate types from your Supabase (`supabase gen types`) and drop them into `types.ts` (we'll maintain this one manually now since Lovable Cloud won't regenerate against your project).

### 5. Switch auth providers on your Supabase
- Enable email/password.
- (Optional) Enable Google OAuth — you'll need to provide a Google OAuth client ID/secret in your Supabase dashboard since Lovable's Google broker only works with Lovable Cloud.

### 6. Verify
- Sign up a fresh test user → confirm profile auto-creates.
- Start a monitoring session → confirm rows land in your Supabase tables.
- Trigger an incident → confirm storage upload works.
- Send password-reset emails to migrated users.

## Technical notes

- **Data export tool**: I'll use the Lovable Cloud `read_query` tool to dump each table as JSON, then bulk-insert into your Supabase via its REST API using the service role key.
- **UUID preservation**: All `id`/`user_id` UUIDs are copied verbatim so FKs stay intact.
- **Storage**: Files are streamed via signed URLs from source bucket → uploaded to destination bucket with the same object paths.
- **RLS during import**: Inserts run as `service_role` so RLS won't block migration.
- **Type drift risk**: Since `types.ts` is normally auto-generated by Lovable Cloud, future schema changes you make in your Supabase dashboard will require manually re-running type generation. The `db.ts` shim already mitigates this.
